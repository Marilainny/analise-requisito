    Conceitos de Análise de Requisitos

    Requisito - funcionalidade que o sistema deve possuir ou uma restrição que deve sastifazer uma necessidade do cliente. Podendo ser dos seguintes tipos:
    Requisito Funcional

  • Recursos do sistema.
  • O que se espera que o sistema faça, independente de como isso será implementado.
    • Exemplo: "A aplicação deve gerar relatórios."
  • O que o sistema não pode fazer.

    • Requisito Não Funcional
  • Qualidade gerais ou restrições do sistema como facilidade de utilização, facilidade de manutenção, segurança, desempenho, dentre outros.
    • Exemplo: "O sistema deve ser baseado em linux."

    Dificuldades na Elicitação dos Requisitos

  • Stakeholders não sabem o que querem do softaware.
  • Stakeholders não sabem explicar o que querem do softaware.
  • Stakeholders podem fazer exigências inviáveis.
  • Stakeholders usam sua própria linguagem.
  • Stakeholders podem ter requisitos conflitantes.
  • Fatores organizacionais podem influênciar como mudança de legislação.
  • Requisitos mudam durante a engenharia de requisitos.

    • Descoberta Dos Requisitos

  • Fontes: documentação, Stakeholders, especificações de sistemas similares.
  • Técnica de extração de requisitos: entrevistas, etnografia (observação do ambiente de trabalho), prototipação, casos de uso.
  • Artefatos gerados: documento de requisito ou especificação de requisitos de software.
  • Casos de usos.
  • Detalhamento dos requisitos.
  • Protótipos.

    • Contexto dos Exemplos dos Artefatos

  • Considere como exemplo nos artefatos um sistema em que o Stakeholder (usário final) tenha pedido um sistema CRM de Vendas que inicialmente será feito somente o cadastro de clientes, de usuários e a autenticação do usuário
  • O cadastro de cliente deve conter os campos CPF, nome e telefone do cliente, que são campos obrigatórios. Deve permitir cadastrar, editar, excluir e consular clientes.
  • O cadastro de usuários deve conter os campos de login (CPF), nome e senha.
  • Deve permitir que o usuário autentique no sistema informando login e senha.
  • Este sistema deve ser online. Deve ser feito na linguagem de programação Java e com banco de dados MySQL.

    Um requisito identifica

    O sistema deve ter um valor ao usuário:

  • Um atributo
  • Uma capacidade
  • Uma característica
  • Uma qualidade

    • Um requisito pode ser

  • Uma descrição de funcionalidade
  • Uma especificação detalhada do comportamento esperado de um sistema
  • Uma propriedade genérica de um sistema
  • Uma restrição técnica do sistema
  • Uma restrição do processo de desenvolvimento
  • Informações sobre uma regra de negócio

    • O manifesto ágil e os princípios ágeis

    • Indivíduos e interação - vem antes de processos e ferramentas
    • Software funcionando - vem antes de documentação abrangente
    • Colaboração do cliente - vem antes de negociação de contrato
    • Resposta à modificação - vem antes de um plano em andamento

    Os princípios Ágeis

    1. A prioridade é satisfazer o cliente por meio de entrega pronta e contínua de valor.
    2. Acolher modificações de requisitos, mesmo no final do desenvolvimento.
    3. Entregar software funcionando com frequência, tempo curto
    4. Interação diárias entre as equipes pessoal do negócio e os desenvolvedores.
    5. Equipe motivada, dar o ambiente e apoio que necessitam e auto-organização.
    6. Reunões com as equipes para evoluir a comunicação.
    7. Software funcionando é a principal medida de progresso.
    8. Processos ágeis e sustentáveis.
    9. Os stakeholders devem manter um constante.
    10. Atenção contínua para a excelência técnica e para um bom projeto aumenta a agilidade.
    11. Simplicidade, maximizar a quantidade de trabalho não realizada.
    12. As melhores arquiteturas, requisitos e projetos surgem de equipes auto-organizadas.
    13. Em intervalos regulares, a equipe reflete sobre como se tornar mais efetiva, depois sintoniza e ajusta o seu comportamento de acordo com as estratégias.

    Modelo de Processo de Software

    Waterfall - separados processos de especificação, desenvolvimento, validação e evolução para depois entregar a versão final.

    • Análise e definição de requisitos do sistema.
    • Design do Software a arquitetura do sistema.
    • Implementação e testes verificando as especificações.
    • Integração e sistema de teste para validação da entrega.
    • Manutenção e aprimoramento do sistema.

    Incremental development - Uma série de de versões são incrementada

    Reuse-oriented software engineering

    Existem três tipos de componentes de software que podem ser usados em um processo:

    1. Web Service.
    2. Coleção de objetos desenvolvidos como um pacote a ser integrado.
    3. Sistemas de software autônomos que são configurados para uso em um determinado meio Ambiente.

    Existem quatro principais atividades requeridas no processo de desenvolvimento.

    1. Estudo de viabilidade - É feita uma estimativa se as necessidades identificadas do usuário podem ser satisfeito usando as tecnologias atuais de software e hardware. O estudo considera se o sistema proposto será rentável do ponto de vista do negócio e se pode ser desenvolvido dentro das restrições orçamentárias existentes. Uma viabilidade estudo deve ser relativamente barato e rápido. O resultado deve informar a decisão de prosseguir ou não com uma análise mais detalhada.
    2. Elicitação e análise de requisitos - Este é o processo de derivação do sistema requisitos através da observação de sistemas existentes, discussões com potenciais usuários e compradores, análise de tarefas e assim por diante. Isso pode envolver o desenvolvimento de um ou mais modelos e protótipos de sistemas. Estes ajudam você a entender o sistema a ser especificado.
    3. Especificação de requisitos - A especificação de requisitos é a atividade de traduzir as informações coletadas durante a atividade de análise em um documento que define um conjunto de requisitos. Dois tipos de requisitos podem ser incluídos neste documento. Os requisitos do usuário são declarações abstratas dos requisitos do sistema para o cliente e o usuário final do sistema; os requisitos do sistema são descrição mais detalhada da funcionalidade a ser fornecida.
    4. Validação de requisitos - Esta atividade verifica os requisitos quanto ao realismo, consistência e integridade. Durante este processo, erros no documento de requisitos são inevitavelmente descobertos. Ele deve então ser modificado para corrigir esses problemas.
